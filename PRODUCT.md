# Product

## Register

product

## Users

**两个主要用户群体，需要分别的界面：**

1. **终端客户**：想要预定演出、赛事、景区、电影等票务的使用者。他们在各种环境下使用（办公室、通勤、家中），需要快速找到心仪的活动并完成预订，期望流程简单、信息清晰、支付安全。

2. **管理员**：负责监察数据、管理用户、维护票务库存的运营人员。他们在办公环境中长时间使用系统，需要高效处理大量信息、快速响应问题、监控系统状态。

## Product Purpose

为 HyperTicket 高性能票务系统构建企业级 Web 前端界面，包含三个核心部分：

- **客户 Web 应用**：让用户能够浏览票务、查看详情、在线预订、管理订单，体验流畅且值得信赖
- **管理后台**：替代 CLI 工具，提供可视化的票务管理、用户管理、数据监控、黑名单控制
- **营销引导页**：展示平台能力和价值，吸引新用户注册使用

技术架构：React + TypeScript + Vite + Ant Design，通过 WebSocket 桥接层连接后端的 TCP + JSON 协议。

成功标准：客户能在 3 步内完成预订，管理员能快速定位和解决问题，界面在日间和夜间模式下都清晰易读。

## Brand Personality

**高端、优雅、无缝**

这是一个企业级票务平台，不是消费级娱乐应用。界面应该传达专业性和可靠性，同时保持现代感和精致感。

- **高端**：精心打磨的细节、克制的配色、考究的排版，让用户感受到平台的品质
- **优雅**：清晰的信息层级、流畅的交互、恰到好处的留白，避免视觉噪音
- **无缝**：操作符合直觉、状态反馈及时、错误处理优雅，用户不需要思考就能完成任务

语气：专业但不冷漠，高效但不生硬。文案直接、准确，避免营销话术。

## Anti-references

**明确不要的方向：**

1. **杂乱无章的传统企业软件**：密集的表格、过时的 UI 控件、混乱的信息层级、缺乏呼吸感的布局。我们要现代、清晰、有节奏的界面。

2. **通用的 SaaS 模板**：千篇一律的渐变、玻璃态效果、装饰性图标、空洞的营销文案（"赋能"、"颠覆"、"生态"）。我们要有辨识度的设计语言，不是从模板库拖出来的页面。

3. **过度装饰的消费级应用**：花哨的动画、夸张的圆角、卡通化的插图。我们服务的是严肃的票务场景，设计应该克制、专业。

## Design Principles

1. **信息优先，装饰克制**  
   每个视觉元素都应该服务于信息传达或操作引导。如果一个设计元素不能帮助用户更快完成任务或更好理解信息，就不应该存在。

2. **为两种用户群体设计两种节奏**  
   客户界面强调引导和信心（大图、清晰的 CTA、简化的选择）；管理界面强调密度和效率（紧凑的表格、快捷操作、批量处理）。不要用同一套模板服务两种需求。

3. **日夜皆宜，不是主题切换的敷衍**  
   日间和夜间模式都应该是精心设计的完整方案，不是简单的颜色反转。对比度、层级、重点在两种模式下都应该清晰。

4. **中文优先的排版**  
   字体选择、行高、字间距都应该为中文阅读优化。避免为英文设计的紧凑排版导致中文拥挤难读。标点、数字、英文混排时保持和谐。

5. **企业级的可靠感**  
   加载状态、错误提示、边界情况都应该有明确的反馈。用户应该始终知道系统在做什么、为什么、接下来会发生什么。不要让用户猜测或等待。

## Accessibility & Inclusion

- **WCAG 2.1 AA 标准**：所有文本对比度 ≥4.5:1（大文本 ≥3:1），交互元素有清晰的焦点状态，表单有明确的标签和错误提示
- **日夜间主题切换**：必需功能，不是可选项。两种模式都应该通过对比度测试
- **键盘导航**：所有操作都可以通过键盘完成，焦点顺序符合逻辑
- **屏幕阅读器支持**：语义化 HTML、ARIA 标签、有意义的替代文本
- **色盲友好**：不仅依赖颜色传达信息，使用图标、文字、位置等多重线索
- **响应式设计**：支持桌面、平板、手机，管理后台优先桌面体验，客户应用全设备优化
- **中文界面**：所有 UI 文案、提示、错误信息使用中文（项目名称 "HyperTicket" 保持英文）
